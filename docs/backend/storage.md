# Storage System

The project uses **MinIO** (an S3-compatible object storage) for managing all binary data.

## Integration Layer

The storage logic is encapsulated in `app/storage/minio.py`.

- **Client**: We use the official `minio` Python SDK.
- **Bucket**: All data is stored in a single bucket defined by `MINIO_BUCKET` (default: `deepagent`).

## Operations

### Presigned URLs
To avoid streaming binary data through the backend, we generate presigned URLs:
- **`get_presigned_upload_url`**: Used by the frontend for direct-to-S3 uploads.
- **`get_presigned_download_url`**: Used for downloading files or rendering images in the UI.

### Internal Transfers
The Backend service interacts directly with MinIO to:
- Stage files for the sandbox executor.
- Retrieve artifacts generated by agent code.
- Manage workspace persistence for the agent.

## Lifecycle & Metadata

While MinIO stores the physical bytes, metadata (filename, content-type, size, owner) is stored in the `thread_files` table in PostgreSQL.

## Sandbox Interaction

The `microsandbox-executor` also interacts with the storage system. When code generates an artifact, the executor uploads it to MinIO and returns the object key to the backend.
